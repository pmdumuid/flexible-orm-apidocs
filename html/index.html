Introduction to flexible-orm

Introduction

Simple PHP 5.3 based ORM allowing cloud-based models and traditional database models to coexist.

It aims to simplify the relationship between the database and the application by presenting database rows in an Object-Oriented fashion.

The ORM package also includes some general application utilities, like the Configuration class for storing application options.

Definitions

The ORM package fulfils the Model part of the Model-View-Controller design pattern. As such, classes that define a single ORM entity (e.g. they extend ORM_Model) are described as models.

Each instance of a model refers to exactly one row in a database and each column a single property. For example if you have the database table (users):

id*name email
1 Jarrod jarrod@suho.com.au
2 Steve steve@suho.com.au
3 Jime jim@suho.com.au

This would be represented by 3 instances of the model User, each of which would have the properties ->id, ->name and ->email.



Initial Setup

To start using the ORM, you will need to create an INI file that includes database details and the location of your packages. If you only have one database, you only need to define the [database] group in the INI file.

Then you simply require the AutoLoader.php file and load your config.

INI File 

 [packages]
 ORM      = "/path/to/ORM"
 MyApp    = "/path/to/MyApp"

 [database]
 name     = "test"
 user     = "root"
 pass     = ""

Then all we need to do is include the ORM in your PHP file and load your settings: 

 require '/path/to/ORM/AutoLoader.php';
 \ORM\Utilities\Configuration::Load('test.ini');
Note:
For more information on how the AutoLoader finds classes, see AutoLoader - Managing Packages.

Using This Documentation

This documentation comes in two parts, the Getting Started tutorial and the API docs.

  • ORM Key API Classes
    • ORM_Model is the basis for the entire ORM
    • ModelCollection simplifies handling groups of objects
    • ORM_Core and ORM_Interface allows you to define an alternative model class that behaves in the same way as ORM_Model (for instance for a non-database backed model).
  • Utilities
    • Configuration in a simple class for centralising option storage
    • PDOFactory handles database prepared statements
    • ORM_PDOStatement expands on the standard PDOStatement class with some convenience methods.
  • Tests
    • ORMTest is the parent class of all unit tests and includes information about running and viewing.
